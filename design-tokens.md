## 1. 全局视觉基调

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   HonestMix 走 暗调专业音频设备 风格。                     │
│                                                         │
│   ── 不是扁平 UI                                        │
│   ── 不是 Material Design                               │
│   ── 是模拟硬件机箱：多层金属边框、铆钉、内凹面板、          │
│       星空背景、呼吸灯光、模拟 VU 表、物理推子手感           │
│                                                         │
│   参考：Softube Tape / FabFilter 系列 / 高端音频硬件面板     │
│                                                         │
│   色彩情绪：暗 → 中性灰黑底 + 冷白文字 + 频谱色渐变做点缀     │
│   材质感：多层描边模拟金属厚度 + 内阴影模拟 CNC 铣削凹槽       │
│   氛围感：星空粒子 + 呼吸灯光 + 径向氛围光                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 色彩体系一句话

**底色是暗的（`#0e0f13`~`#111218`），文字是冷白灰的（`rgba(245,245,240,...)`），强调色来自频谱映射（灰→绿→青），装饰金属件是灰褐色的。**

### 2.4 频谱颜色映射（翻译度 0% → 200%）

> ⚠️ **这不是固定色值，是一个颜色渐变函数。** 推子位置改变时，以下元素同步变色：
> - VU 表填充条底色 + 外发光
> - 曲线面板边框色
> - 顶栏状态指示灯（stDot）色 + 发光
> - 矫正曲线线条色 + 填充色

| 翻译度比例 | R | G | B | 视觉 |
|-----------|----|----|-----|------|
| 0.00（0% 翻译度） | 80 | 78 | 74 | 深灰褐（中性） |
| 0.30（60 翻译度） | 100 | 120 | 80 | 橄灰绿 |
| 0.55（110 翻译度） | 120 | 165 | 75 | 草绿 |
| 0.75（150 翻译度） | 90 | 195 | 70 | 亮绿 |
| 0.90（180 翻译度） | 40 | 210 | 90 | 荧光绿 |
| 1.00（200 翻译度） | 20 | 220 | 110 | 薄荷青 |

> 颜色在两个断点之间做**线性插值**。

> **这是整个 HonestMix 插件最重要的视觉组件 —— 也是用户目光停留最久的地方。**
>
> 原型参考：[ffAudio/Frequalizer](https://github.com/ffAudio/Frequalizer) —— 开源 VST 频谱分析器。
> 其 `FrequalizerEditor.cpp` 的 `createAnalyserPlot()` / `plotFrame()` + `Analyser.h` 的 FFT 管道，
> 映射到本 HTML 的 `CurvePanel` 组件。以下详细记录这条映射链。

---

### 10.0 Frequalizer → HonestMix 映射表

| Frequalizer 源码概念 | HonestMix HTML 实现 | 说明 |
|---------------------|---------------------|------|
| `plotFrame` 网格矩形 | `.curve-panel` + `.curve-grid` | 整个曲线显示区 |
| 10 条竖线 (`drawVerticalLine`) | `linear-gradient(90deg, ...)` 10% 间隔 | 频率分隔线 |
| 3 条 dB 横线 (25% / 50% / 75%) | `background-size: 10% 12.5%` | 8 格横线 = 24dB ÷ 8 = 3dB/格 |
| `maxDB` / `maxDB/2` / `0` / `-maxDB/2` / `-maxDB` | 纵轴标签 +12 / +6 / 0 / −6 / −12 | dB 刻度 |
| `getFrequencyForPosition()` 对数映射 | `f = 20 × 10^(3 × rx)` | 鼠标 X → 频率 |
| 输入=绿 / 输出=红 双曲线对比 | 原始曲线 RAW（灰褐）/ 矫正曲线 CORR（动态色） | 双轨可视化 |
| `createAnalyserPlot()` 生成 `juce::Path` | SVG `<polyline>` / `<polygon>` | 14 个采样点连成曲线 |
| FFT 实时数据 | 硬编码 `rawYs[]` 假数据 | **待替换：真实耳机频响曲线（来自 HeadphoneDatabase）** |

### 10.2 网格系统（GridLines · 07a）

#### 频率轴（横轴 · 对数尺度）

```
X 位置映射（非均匀，对数分布）：
  位置%      频率        备注
  ───────────────────────────────────
   5%        20 Hz       次低频起点
  14%        50 Hz       低音
  24%       100 Hz       低中频
  35%       200 Hz
  46%       500 Hz       中频参考点
  55%         1 kHz      ⭐ 中枢参考频率
  65%         2 kHz      人耳最敏感区
  76%         5 kHz      中高频
  86%        10 kHz      高频
  95%        20 kHz      人耳上限
```

> **频率映射公式**（对数）：`f = 20 × 10^(3 × rx)`  
> 其中 `rx` = 鼠标 X 坐标 ÷ 面板宽度，范围 [0, 1]。
> 验证：`rx=0 → 20Hz` ✓ / `rx=0.5 → 632Hz`（近 1k 对数中点）✓ / `rx=1 → 20kHz` ✓

#### 分贝轴（纵轴 · 线性尺度）

```
Y 位置映射：
  位置%      分贝值      备注
  ───────────────────────────────
   2%       +12 dB      增益上限
  25%        +6 dB
  50%         0 dB      ⭐ 中心参考线（零线/目标线）
  75%        −6 dB
  97%       −12 dB      衰减下限
```

> **分贝映射公式**（线性）：`db = 12 − ry × 24`  
> 其中 `ry` = 鼠标 Y 坐标 ÷ 面板高度，范围 [0, 1]。
> 验证：`ry=0 → +12dB` ✓ / `ry=0.5 → 0dB` ✓ / `ry=1 → -12dB` ✓

#### 网格线样式

| 属性 | 横线（dB 格） | 竖线（频率格） |
|------|-------------|--------------|
| 间距 | 12.5% | 10% |
| 线色 | `rgba(180, 200, 230, 0.04)` | `rgba(180, 200, 230, 0.03)` |
| 线宽 | 1px | 1px |
| 零线（Y=50%） | `rgba(190, 185, 175, 0.15)` — 0.8px 虚线 (dash: 3,4) | — |

### 10.9 数据桥接（引擎 ↔ 仪表盘）

> **这是整份文档最重要的桥接章节。**
> 美工不需要理解 FFT，但需要知道曲线从哪来。工程师需要知道每一步的计算公式。

---

#### 10.9.1 数据源全景

```
┌─────────────────────────────────────────────────────────────────┐
│                    HeadphoneDatabase（引擎）                       │
│                                                                   │
│  getProfileFIR(index)                    getCurveNames()          │
│  → 1024-tap 矫正 FIR 系数                → "Harman OE"            │
│  → 每副耳机一组（1603 组）                 → "Diffuse Field"        │
│                                          → "Flat"                │
│  getProfileName(index)                   → "Free Field"          │
│  → "Audio-Technica ATH-M50x"                                      │
│                                                                   │
└──────────────┬──────────────────────────────┬────────────────────┘
               │                              │
               ▼                              ▼
┌──────────────────────────┐   ┌──────────────────────────┐
│  仪表盘 · RAW 曲线        │   │  仪表盘 · 目标曲线         │
│  = 该耳机原始频响          │   │  = 用户选择的目标曲线      │
│  = 1 / |FFT(FIR)|        │   │  = 水平线（0dB 参考）     │
│                          │   │  或 Harman 等目标曲线      │
└──────────┬───────────────┘   └──────────────────────────┘
           │
           ▼
┌──────────────────────────────────────────────────────────┐
│  仪表盘 · CORR 曲线                                        │
│  = RAW 曲线 × (1 − translationDegree/200)                 │
│    + 目标曲线 × (translationDegree/200)                    │
│                                                          │
│  翻译度 = 0   → CORR = RAW（不矫正，原始耳机声）            │
│  翻译度 = 100 → CORR = (RAW + 目标)/2（半矫正）             │
│  翻译度 = 200 → CORR = 目标（全校正，耳机被完全补偿）        │
└──────────────────────────────────────────────────────────┘
```

---

#### 10.9.2 计算路径（一步一步）

```
步骤 1：用户选耳机
  index = headphoneDB.findProfileByName("AKG K701")
  fir = headphoneDB.getProfileFIR(index)   → float[1024]

步骤 2：FIR → 频响（FFT）
  fft = FFT(fir, 1024)                     → complex[1024]
  magnitude[i] = sqrt(real² + imag²)       → float[1024]
  
  ⚠️ 矫正 FIR 的幅频响应 ≈ 1 / 耳机原始频响
  rawResponse[i] = 1.0 / magnitude[i]      → float[1024]（耳机真实频响）
  
  转 dB：rawDb[i] = 20 × log10(rawResponse[i])

步骤 3：降采样到显示点数
  1024 个 FFT bin → 降采样到 N 个显示点（建议 N = 256）
  
  方法：按对数频率轴分组取平均。
  低频 bin 密集、高频 bin 稀疏 → 对数降采样保持视觉精度。

步骤 4：映射到仪表盘 Y 坐标
  viewBox Y 范围：−1 ~ 132（对应 +12dB ~ −12dB）
  viewBox 中心 Y = 74（对应 0dB）
  
  每个显示点的 Y 坐标：
    y[i] = 74 − (rawDb[i] / 24.0) × 132
  
  验证：rawDb = +12 → y = 74 − (12/24)×132 = 74 − 66 = 8 ✓（接近 SVG 顶部）
        rawDb =   0 → y = 74 − 0 = 74 ✓（SVG 正中）
        rawDb = −12 → y = 74 + 66 = 140 → 夹取到 132 ✓

步骤 5：生成矫正曲线
  用户翻译度 degree（0~200）
  t = degree / 200                         → float [0, 1]
  
  corrY[i] = rawYs[i] + (74 − rawYs[i]) × t
  
  当 t=0（0% 翻译度）→ corr = raw（原始耳机声）
  当 t=0.5（100 翻译度）→ corr = (raw + 74)/2（半矫正）
  当 t=1（200 翻译度）→ corr = 74 水平线（全校正到目标）

步骤 6：渲染
  rawYs → SVG <polyline>（RAW 组）
  corrYs → SVG <polyline>（CORR 组，动态更新）
```

#### 10.9.3 目标曲线处理

```

  "Harman OE" / "Diffuse Field" / "Flat" / "Free Field"

当前 HTML 实现：
  目标 = 水平线 Y=74（对应"Flat"——完全平坦的 0dB 参考线）

如果后续支持非平坦目标曲线（如 Harman OE）：
  1. 工程师从数据库/配置文件获取目标曲线的频响数据（N 个 dB 值）
  2. 映射为 targetYs[] 数组（替换硬编码的 74）
  3. CORR 公式改为：corrY[i] = rawYs[i] + (targetYs[i] − rawYs[i]) × t
  4. 仪表盘上额外画一条目标曲线（ZERO 组 → TARGET 组）
```

#### 10.9.4 当前状态 vs 最终状态

| | HTML 演示（当前） | VST3 实现（最终） |
|------|-----------------|------------------|
| 耳机的原始频响 | 手写 rawYs[14] 假数据 | FFT(FIR) 计算得出 |
| 矫正曲线 | `rawY + (74-rawY) × t` | 公式不变，但 rawY 换成真实值 |
| 目标曲线 | 硬编码水平线 Y=74 | 从数据库读取或默认平坦 |
| 选不同耳机 | 不改变曲线（假数据写死） | **曲线实时切换**到该耳机的真实频响 |
| 选不同目标曲线 | 不生效 | 目标曲线切换，CORR 重新计算 |
| 采样点数 | 14 点 | 256 点（建议） |
| 频率轴精度 | 手动指定的 10 个标签位置 | real FFT bin → log 映射，自动对齐 |

### 10.10 缩放与分辨率（HTML 600px → VST3 自适应）

> **HTML 是为 600px 固定宽度设计的。VST3 插件窗口由 DAW 宿主控制，可能小到 300px、大到 4K 全屏。**
> 直接拉伸会不会变丑？答案：不会——前提是做对了下面三件事。

---

#### 10.10.1 HTML 和 VST3 的本质区别

| | HTML/CSS | VST3/JUCE |
|------|---------|----------|
| 坐标系统 | **像素**（px），绝对值 | **比例**（相对父组件），向量绘制 |
| 缩放行为 | `transform:scale()` → 浏览器重采样 → 可能模糊 | `Graphics::draw*(float x, float y, float w, float h)` → 始终矢量 |
| 圆角 | `border-radius: 10px` — 固定像素 | `drawRoundedRectangle(bounds, radius)` — radius 随组件缩放 |
| 文字 | 固定 `font-size: 9px` | 字体以 point 为单位，DPI 感知 |
| 阴影 | `box-shadow: 0 0 4px rgba(...)` | `DropShadow` 类，blur radius 随比例缩放 |
| 渐变 | CSS gradient 浏览器 GPU 合成 | `ColourGradient` CPU 软件光栅化 |

> **关键结论：CSS 在固定 px 下最美，缩放靠浏览器重采样（有损）。JUCE 天然矢量，缩放无损。**

---

#### 10.10.2 VST3 的缩放机制

引擎已有的缩放方案（来自 `Design.h`）：

```cpp
kRefWidth  = 420          // 设计基准宽（所有像素值以此为参考）
kRefHeightNormal = 300    // 平常态基准高
kRefHeightExpand = 645    // 展开态基准高
kScreenWidthPercent = 28  // 插件宽 = 屏幕宽的 28%
```

**工作原理：**

```
1. 宿主给插件一个窗口尺寸（如 800×500）
2. 引擎计算缩放比：scale = 实际宽度 / kRefWidth
   例：实际 600px → scale = 600/420 ≈ 1.43
3. 所有组件坐标 × scale：
   - 推子轨道 24px → 24 × 1.43 ≈ 34px
   - 推子帽 36×26 → 51×37
   - 曲线面板高度 140px → 140 × 1.43 ≈ 200px
4. 字体：同样按 scale 缩放 → 9px × 1.43 ≈ 13px
```

**所以不管 DAW 给你多大窗口，所有元素等比缩放，比例关系不变，不会变形。**

---

#### 10.10.3 为什么不会"拉伸变丑"

```
担心：推子从 24px 拉到 48px，会不会像素化、锯齿、模糊？

→ 不会。因为：

① 推子轨道不是图片，是 g.fillRoundedRectangle() 画的。
   → 24px 宽是矢量矩形，48px 宽还是矢量矩形。
   → 矢量没有"像素化"这个概念，任意缩放都是清晰的。

② 同理，铆钉是 g.fillEllipse()（矢量圆），
   阴影是 DropShadow（程序生成），
   文字是 FreeType/DirectWrite 渲染（DPI 感知），
   曲线是 juce::Path（贝塞尔/折线）。

③ 唯一的例外：如果你用 JPEG/PNG 图片作为背景或纹理。
   → 这就是为什么星空改用程序生成而不是 JPG。
   → 程序生成的 45 个圆点，不管面板多大，圆永远是圆的。
```

---

#### 10.10.4 两种尺寸下的对比

```
场景 A：小窗口（DAW 侧边栏，~350px 宽）
  scale ≈ 350/420 ≈ 0.83
  推子轨道：24 × 0.83 ≈ 20px（仍然可操作）
  曲线面板：140 × 0.83 ≈ 116px（仍然可读）
  字体缩小但保持清晰
  → 效果：紧凑但不拥挤

场景 B：大窗口（全屏 4K，~800px 宽）
  scale ≈ 800/420 ≈ 1.90
  推子轨道：24 × 1.9 ≈ 46px（非常舒适）
  曲线面板：140 × 1.9 ≈ 266px（惊艳的细节）
  256 个采样点在 266px 面板上显示 → 几乎每个像素一个采样点
  → 效果：仪表盘在 4K 下像是硬件频谱分析仪的屏幕

场景 C：HTML 设计基准（600px）
  scale ≈ 600/420 ≈ 1.43
  推子轨道：24 × 1.43 ≈ 34px
  曲线面板：140 × 1.43 ≈ 200px
  → 最接近 HTML 终版的视觉感受
```

---

#### 10.10.5 HTML 600px vs 引擎 420px —— 需要统一吗？

| | HTML 设计基准 | 引擎设计基准 |
|------|-------------|-------------|
| 基准宽度 | 600px | 420px |
| 基准高度（平常态） | ~328px | 300px |
| 比例 | ~1.83 : 1 | 1.4 : 1 |

> ⚠️ **这两个基准不统一。** 引擎的 420px 和 HTML 的 600px 如果直接用同一份 `scale` 公式，所有像素值会不一致。

**统一方案（二选一）：**

| 方案 | 做法 | 影响 |
|------|------|------|
| **A：引擎迁就 HTML** | 改 `kRefWidth = 600` | 所有令牌值与 HTML 完全一致。组件代码直接写令牌值，不用心算换算。美工和工程师看到的数字一模一样。 |
| **B：HTML 迁就引擎** | 不改代码，但令牌文档标注"引擎基准 = 420，HTML 演示 = 600（×1.43 放大）" | 美工看 600，工程师看 420。两套数字并存，容易出错。 |

> **建议方案 A。** 把 `kRefWidth` 从 420 改成 600，一劳永逸。

---

#### 10.10.6 缩放不会破坏什么（设计师放心清单）

| 你可能担心 | 实际结果 |
|-----------|---------|
| 边框会不会变粗细不均？ | 不会。`drawRoundedRect` 线宽也乘 scale，比例恒定。 |
| 阴影会不会位置错乱？ | 不会。`DropShadow` offset 和 radius 都乘 scale。 |
| 文字会不会发虚？ | 不会。字体引擎按 scale 后的 point size 重新渲染，是原生分辨率。 |
| 圆角会不会变形？ | 不会。`cornerRadius × scale` → 圆角随组件等比缩放。 |
| 曲线会不会出现锯齿？ | 不会。`juce::Path` 是矢量路径，抗锯齿由 GPU/CPU 处理。 |
| 星星会不会变成椭圆？ | 不会。`g.fillEllipse(x, y, w, w)` 正圆始终正圆（w × scale, w × scale）。 |
| 388px 宽 vs 892px 宽，布局会不会乱？ | 不会。FlexBox/Grid 的 `flex: 1 1 0` 是比例分配，不依赖绝对 px。 |

---

#### 10.10.7 需要美工注意的一件事

```
HTML 里你可能写了：
  .fader-track { width: 24px; }

VST3 里等价于：
  faderTrackWidth = 24 × scale;

美工不需要关心 scale 是多少。
你只需要确保设计基准（600px）下所有尺寸是对的。
工程师负责在代码里把每个令牌值乘上 scale。
```

> **一句话：你在 600px 画布上设计，看到的 = 用户最终看到的（比例关系完全一致）。大小由 DAW 窗口决定，美感由令牌决定。**

## 12. 美工交付模板

> 美工按以下格式，每个组件交付一个独立 HTML 文件。

### 模板格式

```html
<!-- ============================================================
  组件：07 · CurvePanel（曲线面板）
  包含子组件：GridLines + CurveLabels + CurveCanvas + StarField
  版本：v1
  日期：2026-07-24
  关联令牌文件：design-tokens.md
  ============================================================ -->

<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <title>HonestMix · 07 · CurvePanel</title>
  <style>
    /* ⚠️ 所有颜色/字号/间距必须引用令牌名（如 --bg-curve-panel） */
    /* 令牌定义在顶部集中声明，方便核对 */
    :root {
      /* ===== 本组件用到的令牌（从 design-tokens.md 复制） ===== */
      --bg-curve-panel:          rgba(0, 0, 0, 0.42);
      --border-curve:            rgba(80, 78, 74, 0.50);
      --border-curve-dim:        rgba(80, 78, 74, 0.15);
      --radius-ml:               8px;
      --shadow-curve-inner:      inset 0 5px 18px rgba(0,0,0,0.60);
      --shadow-curve-top-rim:    0 2px 0 rgba(255,255,255,0.01);
      /* ... 本组件用到的所有令牌 ... */
    }

    /* ===== 组件样式（只引用 var() 不写硬编码值） ===== */
    .curve-panel {
      background: var(--bg-curve-panel);
      border: 2px solid var(--border-curve);
      border-radius: var(--radius-ml);
      box-shadow: var(--shadow-curve-inner), var(--shadow-curve-top-rim);
    }

    /* ===== 状态变体 ===== */
    .curve-panel.dim { border-color: var(--border-curve-dim); }
    .curve-panel.active { border-color: var(--border-curve-dynamic); }
    /* 动态色需标注：此色由 spectrumColor 函数计算，不写死 */
  </style>
</head>
<body>

  <!-- ===== 默认态 ===== -->
  <div class="curve-panel" data-component="CurvePanel" data-state="default">
    <!-- ... 子组件 ... -->
  </div>

  <!-- ===== Dim 态 ===== -->
  <div class="curve-panel dim" data-component="CurvePanel" data-state="dim">
    <!-- ... 子组件 ... -->
  </div>

  <!-- ===== Active 态 ===== -->
  <div class="curve-panel active" data-component="CurvePanel" data-state="active">
    <!-- ... 子组件 ... -->
  </div>

  <!-- ===== 说明注释 ===== -->
  <!--
    状态切换时机：
    - default: 无交互
    - dim:     点击第 1 次 / hover 时（边框变暗）
    - active:  拖拽推子改变翻译度时（边框色随翻译度变）

    动态色处理：
    --border-curve-dynamic 不是固定令牌，由 spectrumColor(translationDegree/200) 生成
  -->

</body>
</html>
```
